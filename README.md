### Ответ на задание 1

Главное отличие между этими режимами заключается в том, кто имеет право изменять данные и как система справляется с нагрузкой на запись.

Master-Slave (Ведущий — Ведомый)

В этом режиме все роли четко разделены: один сервер является главным (Master), а остальные — подчиненными (Slaves). 
Как работает: Все операции записи (INSERT, UPDATE, DELETE) идут строго на Master-сервер. Slave-серверы копируют эти изменения и используются только для чтения (SELECT).
Преимущества:
- Простота: Нет риска конфликтов данных, так как источник изменений только один.
- Масштабируемость чтения: Можно добавить сколько угодно Slave-серверов для обработки огромного количества запросов на чтение.
Недостатки:
 - Ограничение записи: Запись нельзя масштабировать добавлением новых серверов — Master остается «узким местом».
 - Единая точка отказа: Если Master выйдет из строя, запись в базу прекратится, пока один из Slave не будет назначен новым Master (часто это требует ручного вмешательства). 

Master-Master (Multi-Master)

В этом режиме все серверы в кластере равноправны и каждый может принимать любые запросы. 
Как работает: Вы можете записывать данные на любой из Master-серверов. Они синхронизируют изменения между собой, чтобы данные везде были одинаковыми.
Преимущества:
 - Масштабируемость записи: Нагрузка на запись распределяется между несколькими узлами.
 - Высокая доступность: Если один Master упадет, остальные продолжат принимать и чтение, и запись без пауз в работе системы.
Недостатки:
 - Конфликты данных: Если два пользователя одновременно изменят одну и ту же строку на разных Master-серверах, системе придется решать, чья версия «правильная» (Conflict Resolution).
 - Сложность настройки: Такая архитектура гораздо сложнее в поддержке и синхронизации.

Резюмируя выше сказанное:
 - Master‑slave — оптимальный выбор для большинства веб‑приложений, где операций чтения в разы больше, чем записи. Проще в настройке, надёжнее в плане консистентности данных.
 - Master‑master — подходит для систем с жёсткими требованиями к доступности и распределённым архитектурой. Требует тщательной проработки логики разрешения конфликтов и более сложного администрирования.
